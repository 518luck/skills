# AGENTS.md 生成器

生成符合项目真实约定的 `AGENTS.md`，把用户提供的概念和问题现象改写成 AI 可执行规则，并在子目录文件中只保留局部新增、局部覆盖和局部命令，避免重复父级规则。

## 使用场景

- 当用户要求“生成 AGENTS.md”“写 AI agent 项目规则”“整理 AI 编码规范”“创建子目录 AGENTS.md”时使用。
- 当用户要求把已有项目规则整理为可被 AI 编码工具遵循的说明文件时使用。
- 当用户说“AI 总是不遵守某规范”“把这个概念写进 AGENTS.md”“补充一条 AI 规则”或提供文档片段、口头经验、问题现象时使用。
- 当用户要求为 monorepo、前后端分离、测试目录、模块目录等创建局部 `AGENTS.md` 时使用。
- 不要用模板凭空生成项目规则；必须先读取项目文件和已有规则。
- 不要在用户只要求解释 AGENTS.md 标准、比较工具兼容性或写普通文档时自动创建文件。

## 工作流程

1. 确认目标路径和范围。
   - 如果用户没有指定路径，默认目标是仓库根目录的 `AGENTS.md`。
   - 如果用户指定目录，目标是该目录下的 `AGENTS.md`，不是小写 `agents.md` 或 `Agent.md`。
   - 如果用户只要求输出内容而不是写文件，只输出 Markdown，不修改文件。
2. 收集根目录上下文。
   - 读取已有 `AGENTS.md`，如果存在则以维护和改进为主，不要无故重写。
   - 读取能证明项目约定的文件：`README*`、`CONTRIBUTING*`、`package.json`、锁文件、`Makefile`、`justfile`、`pyproject.toml`、`Cargo.toml`、`go.mod`、CI 配置、测试配置、lint/formatter 配置、PR 模板和关键文档。
   - 从真实配置中提取包管理器、运行时、默认命令、测试命令、lint 命令、类型检查命令、构建命令、提交/PR 规则和安全限制。
3. 收集子目录上下文。
   - 先解析目标目录相对仓库根目录的位置。
   - 从仓库根目录到目标目录父级，按顺序查找并读取所有已存在的祖先 `AGENTS.md`。
   - 如果目标目录已有 `AGENTS.md`，也要读取并作为更新基础。
   - 读取目标目录内的 `README*`、局部配置文件、测试文件、入口文件、包配置、目录结构和邻近源码，提取只适用于该目录的规则。
4. 将用户概念转成 AI 规范。
   - 把用户的抽象概念、文档说明、问题现象或反复纠正内容拆成具体触发条件、应做行为、禁止行为、验证方式和适用范围。
   - 如果用户描述的是 AI 常犯错，优先写成 `Never`、`Always` 或 `When...` 规则，而不是解释性段落。
   - 如果用户给的是长文档，只提取会影响 AI 行为的项目特定规则；不要把文档原文搬进 `AGENTS.md`。
   - 如果无法判断规则应放在根目录还是子目录，先根据目标文件和影响范围推断；仍不确定时提问。
5. 建立继承与去重表。
   - 把祖先规则按主题归类：AI 行为、环境、命令、代码风格、测试、架构、安全、提交/PR。
   - 标记每条候选规则为 `继承`、`局部补充`、`局部覆盖` 或 `冲突待确认`。
   - 子目录 `AGENTS.md` 只写 `局部补充` 和 `局部覆盖`；不要重复 `继承` 规则。
   - 如果局部规则只是父级命令加路径参数，写清路径差异，不要复制完整父级章节。
   - 如果局部规则与父级冲突，除非项目文件有明确证据，否则先询问用户。
6. 生成或更新文件。
   - 根目录文件写全局事实、全局命令和所有目录都应遵守的规则。
   - 子目录文件写适用范围、继承说明、局部结构、局部命令、局部代码规则、局部测试规则和必要覆盖规则。
   - 补充现有 `AGENTS.md` 时，把新规则放进最相关章节；不要为了加入一条规则重排整个文件。
   - 优先使用简洁列表和可执行命令；避免长段落和抽象口号。
7. 验证结果。
   - 检查文件名必须是 `AGENTS.md`。
   - 检查内容没有占位符、虚构命令或无法从项目文件或用户明确说明推导出的规则。
   - 检查用户提供的概念已经被改写为可执行、可验证的 AI 行为规范。
   - 检查子目录文件没有重复父级/爷爷级已有规则。
   - 如果修改了已有文件，确认没有删除仍然有效的项目特定规则。

## 概念转 AI 规范

用户给出的内容经常是概念、背景文档或对 AI 行为的抱怨。写入 `AGENTS.md` 前，必须先转成面向 AI 的行为规则。

转换维度：

- 触发条件：AI 在什么场景下需要遵守。
- 应做行为：AI 应该具体执行什么动作。
- 禁止行为：AI 不应该做什么。
- 判断标准：如何验证 AI 是否遵守。
- 适用范围：这是根目录全局规则，还是某个子目录局部规则。

改写模板：

```markdown
用户原话：<用户提供的概念、文档片段或问题现象>
规范化规则：<Always / Never / Prefer / Avoid / When... 形式的可执行规则>
适用范围：<全局 | 子目录路径>
放置章节：<AI 行为指令 | 代码风格 | 测试规则 | 架构约束 | 注意事项 | 其他章节>
验证方式：<如何判断 AI 已遵守>
```

写入原则：

- 优先把规则写成短句列表，每条只约束一个行为。
- 用 `Always` 表示强制动作，用 `Never` 表示禁止行为，用 `Prefer` 表示默认偏好，用 `Avoid` 表示不推荐做法，用 `When...` 表示场景化规则。
- 如果规则是为了防止 AI 重复犯错，优先放到 `AI 行为指令` 或 `注意事项`。
- 如果规则涉及命名、依赖、目录边界、测试方式、架构约束或安全要求，放到对应主题章节。
- 如果用户原话包含背景解释，只保留会改变 AI 行为的部分。

转换示例：

```markdown
用户说：AI 老是改无关文件。

写成规范：
- 聚焦当前任务，只修改完成用户请求所必需的文件。
- 不要进行无关重构、格式化或目录整理。
- 如果发现邻近代码有问题但不影响当前任务，只在最终说明中指出，不要直接修改。
```

```markdown
用户说：这里的接口概念比较复杂，要先看文档。

写成规范：
- 修改接口行为前，必须先阅读本目录的 `README.md` 和相关 API 文档。
- 不要仅凭函数名或类型定义推断业务语义。
- 如果文档与代码行为冲突，先说明冲突并询问用户，不要自行选择。
```

```markdown
用户说：这套组件有自己的设计语言，不要做成普通后台页面。

写成规范：
- 修改本目录 UI 前，先查看相邻页面和组件，延续已有视觉语言、间距、色彩和交互模式。
- 不要引入与现有设计系统冲突的通用后台布局。
- 新增组件时，优先复用本目录已有组件和样式 token。
```

## 子目录模板

子目录 `AGENTS.md` 必须明确继承上级文件，并只记录本目录差异：

```markdown
# <目录名> — AI Agent 局部指南

> 本文件继承上级 `AGENTS.md`，仅记录本目录的额外规则、覆盖规则和局部命令。

## 适用范围

- 适用于 `<target-path>/**`。
- 上级规则仍然有效，除非本文件明确覆盖。

## 继承说明

- 已继承：<简要列出父级中仍适用的主题，不复制具体规则>。
- 本文件只补充：<局部主题>。

## 本目录结构

- `<path>/`：<用途>
- `<path>/`：<用途>

## 局部命令

- 运行本目录测试：`<exact command>`
- 运行本目录 lint/typecheck：`<exact command>`

## 局部代码规则

- <只适用于本目录的命名、架构、依赖、文件组织或框架规则。>

## 局部测试规则

- <只适用于本目录的测试位置、夹具、运行方式或覆盖要求。>

## 覆盖父级规则

- 本目录覆盖父级规则：<父级规则> → <本目录规则>。

## 注意事项

- <本目录容易出错的事实。>
```

## 规则

- 必须使用标准 Markdown，不要添加 YAML frontmatter 或客户端专有字段。
- 必须使用全大写文件名 `AGENTS.md`。
- 必须优先写具体、可验证、可执行的规则，例如完整命令、路径、配置文件和检查步骤。
- 必须从项目真实文件或用户明确说明中提取规则；不能因为模板里有某章节就编造内容。
- 根目录文件只写全局规则；子目录文件只写局部差异。
- 不要重复父级、爷爷级或更高层 `AGENTS.md` 已有的通用规则。
- 不要写“写好代码”“遵循最佳实践”“确保测试通过”这类不可验证的空泛规则。
- 不要建议跳过、删除或注释掉测试。
- 不要建议使用 `@ts-ignore`、`@ts-expect-error`、`eslint-disable` 或等价手段掩盖问题，除非项目已有明确允许条件并写明审批/说明要求。
- 不要把完整外部文档、长规范或大量示例塞进 `AGENTS.md`；过长内容应转为项目文档并在 `AGENTS.md` 中引用。

## 验证

- 对根目录文件，检查是否覆盖项目概览、关键命令、代码规则、测试/质量检查和项目特定注意事项。
- 如果用户提供了概念、文档片段或问题现象，检查它们是否已转成触发条件、应做行为、禁止行为和验证方式。
- 对子目录文件，检查是否读取了目标目录链路上的所有祖先 `AGENTS.md`。
- 对子目录文件，逐条比对父级规则，删除语义重复内容。
- 检查所有命令都包含明确工作目录或可从上下文判断的执行位置。
- 检查没有保留 `<占位符>`、`[待补充]`、`TODO` 等未完成内容。
- 如果验证失败，先修正文件，再重新检查。

## 注意事项

- `AGENTS.md` 是开放 Markdown 格式，没有固定 schema；结构应服务于项目可读性和 AI 可执行性。
- 距离编辑文件最近的 `AGENTS.md` 通常优先级最高；因此子目录规则应更具体，而不是复制全局规则。
- 用户消息中的即时要求优先于 `AGENTS.md`；不要在文件中写与用户明确要求冲突的内容。
- 大型项目应控制根目录文件长度，根目录保留全局导航和通用约束，领域细节下沉到子目录 `AGENTS.md`。
- 如果父级规则已经过时，不要在子目录里隐式修正；应提醒用户需要更新父级文件。
