# 片段拼装规则

本文件处理使用 agents-snippets 片段库拼装项目 AGENTS.md 的任务。

## 前置判断：是否适合片段拼装

在执行拼装前，先判断目标 AGENTS.md 的性质：

适合片段拼装的信号：

- 项目刚初始化，还没有 AGENTS.md。
- 用户明确说"帮我用片段组装"或"帮我初始化 AGENTS.md"。
- 项目使用了片段库中已有的基础设施（Redis、PG 等）。
- 用户要求标准化、规范化。

不适合片段拼装的信号：

- AGENTS.md 内容高度绑定特定业务逻辑，无通用性。
- 项目有独特的架构模式，片段库中没有对应片段。
- 用户说"按项目实际情况写"或"不要用模板"。
- 维护已有的、未使用片段拼装的 AGENTS.md，且用户没有要求迁移。

判断结果：

- 适合 → 执行本文件的拼装工作流。
- 不适合 → 交给原有任务路由（根目录/子目录/维护）。
- 不确定 → 问用户一个简短问题："这个项目是否适合使用标准片段组装，还是需要按实际情况定制？"

## AGENTS.md 适用性分类

拼装时需要识别目标 AGENTS.md 的类型：

| 类型 | 特征 | 处理方式 |
|------|------|----------|
| **可解耦型** | 包含多个通用基础设施/架构/风格规则 | 片段拼装 + 项目特有规则 |
| **项目特有型** | 内容高度绑定项目业务逻辑 | 不拼装，走原有流程 |
| **混合型** | 部分通用 + 部分项目特有 | 通用部分引用片段，特有部分单独写 |

## 拼装工作流

### 第一步：收集项目上下文

读取以下文件判断项目特征：

1. `package.json`、`go.mod`、`pyproject.toml`、`Cargo.toml` 等 — 判断语言、运行时、依赖。
2. 目录结构 — 判断架构模式（是否有 `src/`、`app/`、`prisma/` 等）。
3. 已有配置文件 — 判断工具链（lint、test、CI）。
4. 已有的 `AGENTS.md` — 如果存在，判断是否适合迁移到片段拼装。

### 第二步：匹配片段

根据项目特征匹配片段库中的文件：

1. 确定片段库位置：与 skill 同仓库的 `.well-known/agents-snippets/`。
2. 按以下顺序匹配：

基础设施匹配（`infrastructure/`）：

- 检测到 Prisma → 匹配 `postgresql/_base.md` + `postgresql/ts-prisma.md`
- 检测到 ioredis / go-redis / jedis → 匹配 `redis/_base.md` + 对应语言+包文件
- 检测到 Resend / nodemailer 等 → 匹配 `email/_base.md` + 对应语言+包文件
- 其他服务同理

架构匹配（`architecture/`）：

- 检测到 Next.js App Router + `src/` 目录 → 匹配 `nextjs-thin-route.md`
- 检测到 FSD 目录结构 → 匹配 `fsd/_base.md` + `fsd/react.md`
- 其他模式同理

风格匹配（`style/`）：

- 检测到 TypeScript → 匹配 `typescript.md`
- 检测到 Go → 匹配 `go.md`（如果存在）
- 通用规范始终匹配：`naming.md`、`comment.md`

匹配规则：

- 每个服务目录先匹配 `_base.md`，再匹配语言+包文件。
- 如果语言+包文件不存在（片段库尚未覆盖），只用 `_base.md` 并在输出中说明缺失。
- 架构和风格如果没有匹配项，跳过，不要强行套用。

### 第三步：读取并替换变量

1. 读取所有匹配的片段文件。
2. 将片段中的 `{变量}` 替换为从项目文件中提取的实际值。
3. 无法自动确定的变量保留为 `{变量名}` 并在输出中列出，提醒用户手动补充。

常用变量自动提取逻辑：

| 变量 | 提取来源 |
|------|----------|
| `{schema_dir}` | 检测 `prisma/schema/schema.prisma` 是否存在 |
| `{generate_cmd}` | 读取 `package.json` 的 scripts |
| `{migrate_cmd}` | 读取 `package.json` 的 scripts |
| `{infrastructure_dir}` | 检测 `src/shared/lib/infrastructure` 或类似目录 |
| `{pkg_manager}` | 检测锁文件（pnpm-lock.yaml → pnpm 等） |
| `{typecheck_cmd}` | 读取 `package.json` 的 scripts |
| `{lint_cmd}` | 读取 `package.json` 的 scripts |

### 第四步：拼装

拼装顺序：

```markdown
# {项目名} — AI Agent 开发指南

## 项目概览
（从项目文件提取的 1-3 句描述）

## 项目结构
（从实际目录结构生成）

[基础设施片段 — 按服务排列]
## Redis
（redis/_base.md 内容）
（redis/ts-ioredis.md 内容）

## PostgreSQL
（postgresql/_base.md 内容）
（postgresql/ts-prisma.md 内容）

[架构片段]
## Next.js 路由层
（nextjs-thin-route.md 内容）

[风格片段]
## 代码风格
（typescript.md 内容）
（naming.md 内容）

[项目特有规则]
## 项目特定规则
（从项目文件提取的、片段库无法覆盖的规则）

## 验证命令
（从项目 scripts 提取）

## 子目录 AGENTS.md
（导航，如果项目有子目录 AGENTS.md）
```

### 第五步：输出

拼装完成后输出：

```markdown
已拼装：`<path>/AGENTS.md`
匹配片段：<列出所有匹配的片段文件路径>
未匹配：<列出检测到但片段库中未覆盖的服务/模式>
未解析变量：<列出无法自动替换的变量及建议值>
项目特有规则：<说明哪些内容是片段库无法覆盖的>
```

## 拼装规则

- 片段内容按原样拼装，不要改写片段的措辞或结构。
- 多个片段之间用一个空行分隔，不要添加额外分隔线。
- 片段的标题层级可能需要调整：如果片段内部用 `#`，拼装到 AGENTS.md 时降为 `##` 或 `###`。
- 变量替换只替换能确定的值；不确定的保留 `{变量名}` 并在输出中说明。
- 如果项目特有规则与片段规则冲突，以项目特有规则为准，并在项目特有规则章节说明覆盖关系。

## 部分拼装

用户可能只需要拼装部分片段（如"只要 Redis 和 PG 的规则"）：

- 只拼装用户指定的片段。
- 未指定的片段不拼装，不提示，不报错。
- 如果用户指定了片段库中不存在的片段，提示并跳过。

## 混合型处理

当项目既有通用规则又有项目特有规则时：

1. 通用规则部分使用片段拼装。
2. 项目特有部分从项目文件提取或由用户补充。
3. 两部分在 AGENTS.md 中明确分界：

```markdown
<!-- 以下规则来自片段库 -->
## Redis
...

<!-- 以下为项目特有规则 -->
## 项目特定规则
...
```

不使用 HTML 注释也可以，但必须在拼装输出中说明哪些是片段、哪些是项目特有。

## 验证清单

- 片段库路径存在且可访问。
- 所有匹配的 `_base.md` 已包含。
- 语言+包文件与项目实际依赖一致。
- 变量已替换为实际值或明确标注为待补充。
- 没有拼装不相关的片段。
- 项目特有规则与片段规则不重复。
- 拼装后的 AGENTS.md 结构连贯，标题层级正确。
- 没有残留的 `{变量名}` 占位符（或已在输出中说明）。
