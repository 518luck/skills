# AGENTS.md Writer

生成和维护项目 `AGENTS.md` 的入口 skill。具体规则按任务类型从 `references/` 按需读取，避免把所有细节塞进主上下文。支持使用片段库（`.well-known/agents-snippets/`）进行模块化拼装。

## 使用场景

- 当用户要求生成根目录 `AGENTS.md`、编写 AI agent 项目规则或整理项目级 AI 编码规范时使用。
- 当用户要求生成某个子目录的 `AGENTS.md`，并需要继承父级、爷爷级等链路规则且避免重复时使用。
- 当用户要求维护、补充、修正已有 `AGENTS.md` 时使用。
- 当用户提供概念、文档片段、口头经验或"AI 总是不遵守某规范"的问题现象，希望转成 AI 可执行规则时使用。
- 当用户要求使用片段组装 AGENTS.md、初始化项目规则或标准化项目规范时使用。
- 不要在用户只要求解释 AGENTS.md 标准、比较工具兼容性或写普通文档时自动创建文件。

## 前置判断：选择生成模式

在执行任务路由前，先判断目标 AGENTS.md 适合哪种生成模式：

**适合片段拼装的信号：**

- 项目刚初始化，还没有 AGENTS.md。
- 用户明确说"帮我用片段组装""帮我初始化""标准化"。
- 项目使用了片段库中已有的基础设施（Redis、PG 等）。

**不适合片段拼装的信号：**

- AGENTS.md 内容高度绑定特定业务逻辑，无通用性。
- 项目有独特的架构模式，片段库中没有对应片段。
- 用户说"按项目实际情况写"或"不要用模板"。
- 维护已有的、未使用片段拼装的 AGENTS.md，且用户没有要求迁移。

**判断结果：**

- 适合片段拼装 → 进入"片段拼装"任务路由。
- 不适合 → 进入原有任务路由（根目录/子目录/维护）。
- 不确定 → 问用户："这个项目是否适合使用标准片段组装，还是需要按实际情况定制？"

## 任务路由

先根据前置判断选择模式，再读取对应参考文件。

### 片段拼装模式

- 片段拼装生成 AGENTS.md：读取 `references/snippets-assembler.md`。
- 片段拼装 + 维护已有文件：同时读取 `references/snippets-assembler.md` 和 `references/maintain-agents.md`。

### 原有模式

- 生成或重写仓库根目录 `AGENTS.md`：读取 `references/root-agents.md`。
- 生成或重写子目录 `AGENTS.md`：读取 `references/nested-agents.md`。
- 维护、补充或修正已有 `AGENTS.md`：读取 `references/maintain-agents.md`。
- 维护子目录 `AGENTS.md`：同时读取 `references/nested-agents.md` 和 `references/maintain-agents.md`。
- 用户给出概念、文档片段或 AI 常犯错：读取 `references/maintain-agents.md`，先转成 AI 行为规范，再写入目标文件。

如果用户没有指定目标路径，默认处理仓库根目录的 `AGENTS.md`。如果无法判断是根目录、子目录还是维护任务，先问一个简短问题确认。

## 通用工作流程

1. 确认目标路径、任务类型和是否需要实际写文件。
2. 执行前置判断，选择生成模式。
3. 按"任务路由"读取对应 reference，不要读取无关参考文件。
4. 读取现有 `AGENTS.md` 和必要项目文件，先收集真实上下文再生成规则。
5. 只写能从项目文件或用户明确说明推导出的规则，不要凭模板编造命令或约定。
6. 修改文件时保持最小变更；维护已有文件时不要为了补一条规则重排全文。
7. 写入后按对应 reference 的验证清单检查。

## 通用规则

- 必须使用全大写文件名 `AGENTS.md`。
- `AGENTS.md` 使用标准 Markdown，不要添加 YAML frontmatter 或客户端专有字段。
- 规则必须具体、可执行、可验证；不要写"写好代码""遵循最佳实践""确保测试通过"这类空泛表述。
- 子目录 `AGENTS.md` 只写局部补充、局部覆盖和局部命令，不重复父级规则。
- 维护规则时，必须把用户的概念或问题现象转成面向 AI 的行为约束，而不是原样粘贴文档。
- 维护过程中如果发现通用度高的规则，建议用户考虑提取到片段库，但不要自动执行。
- 不要建议跳过、删除或注释掉测试。
- 不要建议用 `@ts-ignore`、`@ts-expect-error`、`eslint-disable` 或等价手段掩盖问题，除非项目已有明确允许条件并写明审批或说明要求。

## 输出格式

完成写入后输出：

```markdown
已生成：`<path>/AGENTS.md`
生成模式：<片段拼装 | 原有生成 | 维护更新>
任务类型：<根目录生成 | 子目录生成 | 片段拼装 | 维护更新>
参考规则：<读取的 references 文件>
依据：<读取的关键项目文件或用户明确说明>
片段匹配：<匹配的片段文件列表，或"不适用">
规范化：<如果用户提供了概念、文档片段或问题现象，说明已如何转成 AI 规则；否则写"不适用">
去重：<子目录或维护场景说明避免重复了哪些父级/已有规则；不适用时写"不适用">
验证：<已检查的项目，或无法验证的原因>
```

只输出草稿时输出：

````markdown
```markdown
<完整的 AGENTS.md 内容>
```
````

## 验证

- 检查前置判断是否已执行。
- 检查已读取与任务类型和生成模式匹配的 reference 文件。
- 检查目标文件名是 `AGENTS.md`。
- 检查内容没有保留 `<占位符>`、`[待补充]`、`TODO` 等未完成内容。
- 检查所有命令都能从项目文件或用户明确说明中找到依据。
- 检查维护或子目录场景没有重复已有规则。
- 如果是片段拼装模式，检查片段匹配结果和变量替换是否完整。
