# Git 提交与推送

安全地检查、暂存并提交 Git 变更，必要时按用户明确要求推送。

## 使用场景

- 当用户要求“提交代码”“帮我 commit”“生成提交信息”“commit and push”或“推送本地变更”时使用。
- 当用户只要求生成提交信息时，只输出提交信息，不执行 `git commit`。
- 当用户只要求查看状态、解释 diff、代码审查或排查问题时，不要提交或推送。
- 默认只提交；只有用户明确要求 push、推送、发布或同步远端时才执行 `git push`。

## 工作流程

1. 收集 Git 上下文。
   - 必须检查 `git status --short`、`git diff`、`git diff --cached` 和 `git log --oneline -10`。
   - 如果动态上下文没有自动展开，手动运行这些命令获取同等信息。
2. 判断提交范围。
   - 只处理用户要求的变更或本次任务产生的变更。
   - 如果存在无关文件、用户未说明的修改、疑似密钥或临时文件，不要暂存它们。
   - 如果已有 staged 变更，先确认它们是否属于本次提交；不要随意取消暂存或覆盖用户的暂存区。
3. 按变更类型拆分提交。
   - 每个提交只包含同一类型的变更。
   - 如果不同文件属于不同目的，拆成多个提交。
   - 每个提交最多覆盖 3 个清晰相关的问题。
4. 执行必要验证。
   - 优先运行与变更相关的现有测试、格式化或 lint 命令。
   - 如果没有合适命令或用户要求跳过验证，在最终说明中写明。
5. 暂存文件。
   - 使用路径级别的 `git add <path>`，只暂存目标文件。
   - 避免无差别使用 `git add .`，除非已确认所有变更都属于本次提交。
6. 创建提交。
   - 使用中文约定式提交信息。
   - 提交前再次检查 `git status --short` 或 `git diff --cached`，确认 staged 内容正确。
7. 处理失败。
   - 如果 hook、lint 或测试失败，先阅读错误并修复，再重新验证和提交。
   - 不要使用 `--no-verify` 跳过 hook，除非用户明确要求。
   - 不要 amend 已创建的提交，除非用户明确要求。
8. 推送。
   - 只有用户明确要求推送时，才运行 `git push`。
   - 推送前检查当前分支和上游分支；如果没有 upstream，先说明将要设置的远端和分支。

## 提交信息格式

```
<type>(<scope>): <subject>
```

- `scope` 可省略；只有在项目已有明确模块名或变更范围清晰时使用。
- `subject` 必须使用中文，用具体、自然的语言说明改了什么。
- 不要写笼统信息，例如 `fix: 修复问题`、`chore: 更新代码`。
- 优先匹配最近提交的风格，但不要违反本 skill 的安全规则。

示例：

```text
feat(report): 增加弹窗生成规则
fix(auth): 修复登录态过期判断
docs: 补充 skill 制作规范
chore: 更新依赖锁文件
```

## 允许的类型

| 类型     | 含义                                   |
| -------- | -------------------------------------- |
| feat     | 新功能                                 |
| fix      | 问题修复                               |
| docs     | 仅文档变更                             |
| style    | 格式调整（空格、分号等），无逻辑变更   |
| refactor | 代码重构，既不是修复也不是新增功能     |
| perf     | 性能优化                               |
| test     | 添加或更新测试                         |
| chore    | 杂项                                   |
| build    | 构建系统变更（webpack、gulp、npm 等）  |
| ci       | CI/CD 配置变更（chart、Dockerfile 等） |
| revert   | 回滚之前的提交                         |

## 规则

- 必须先检查工作区和暂存区，再决定提交范围。
- 必须只暂存目标文件，不要提交无关改动。
- 必须避免提交密钥、令牌、`.env`、本地缓存、构建产物或调试临时文件，除非用户明确要求且项目约定需要。
- 不要因为用户说“提交”就自动推送。
- 不要修改 git config、跳过 hooks、强推、创建空提交或使用交互式 git 命令，除非用户明确要求。
- 如果仓库已有提交信息规范，以仓库规范优先；若与本 skill 冲突，先说明冲突并采用仓库规范。

## 输出格式

提交完成后输出：

```markdown
已提交：`<commit-hash>` `<commit-message>`
验证：<已运行的验证命令，或说明未运行原因>
推送：<已推送到 remote/branch，或说明未推送>
```

只生成提交信息时输出：

```markdown
<type>(<scope>): <中文 subject>
```

## 验证

- 检查 `git status --short`，确认提交后的工作区状态符合预期。
- 如果执行了推送，确认 `git push` 成功且没有推送到错误分支。
- 如果仍有未提交文件，说明它们为什么保留未提交。
- 如果验证失败，修复后重新运行验证；无法修复时说明失败命令和原因。

## 动态上下文

支持动态上下文注入的客户端可使用以下内容作为初始 Git 上下文。不支持时，手动运行相同命令。

### GIT STATUS

!`git status --short`

### GIT DIFF

!`git diff`

### GIT DIFF --CACHED

!`git diff --cached`

### RECENT COMMITS

!`git log --oneline -10`

## 注意事项

- 用户的未提交改动可能与当前任务无关，不能为了“干净工作区”而回滚或改写。
- 如果同一个文件同时包含目标变更和无关变更，使用可控方式只提交相关部分；无法安全拆分时先询问用户。
- 如果远端推送被拒绝，不要自动 rebase、merge 或 force push；先说明原因并询问下一步。
