# Redis 通用规则

## Key 命名

- 使用 `{业务域}:{实体}:{标识}` 格式，例如 `user:session:abc123`、`cache:spec:detail:42`。
- 禁止使用无前缀的 key。
- 前缀和字段之间使用 `:` 分隔。

## TTL

- 当 Redis 作为缓存层使用时，所有 key 必须设置 TTL，不允许永久 key。
- 当 Redis 作为主数据存储时（排行榜、计数器、延迟队列等），可以不设 TTL，但必须在 key 命名中区分用途（如用 `db:` 前缀），避免与缓存 key 混淆。
- Session 类：按认证策略设定，通常 24h。
- Cache 类：按业务场景设定，默认 5min。
- Rate Limit 类：按限流窗口设定。

## 连接管理

- 连接和操作必须统一通过适配层访问，禁止在业务代码中直接初始化客户端或使用原始 SDK API。
- 适配层负责连接池管理、重试策略和错误处理。
- 业务代码只调用适配层暴露的语义化函数。

## 序列化

- 存储复杂数据时使用 JSON 序列化。
- 简单计数器、标志位等可以使用原生类型。
- 不要存储过大的 value（单 key 建议 < 1MB）。

## 禁止事项

- 禁止 `KEYS *` 命令在生产环境使用；使用 `SCAN` 替代。
- 禁止在业务代码中硬编码 key 名；使用常量或 key builder 函数。
- 禁止在循环中单次执行命令；使用 Pipeline 或批量操作。

## 变量

无额外变量。适配层路径在语言+包文件中通过 `{infrastructure_dir}` 指定。
