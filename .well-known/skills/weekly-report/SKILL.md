# 周报生成器

根据项目最近 7 天的 Git 提交记录，自动生成周报表格。

## 工作流程

### 第 1 步：收集提交数据

运行以下命令获取最近 7 天的完整提交信息：

```bash
git log --since="7 days ago" --until="now" --all --no-merges \
  --format="%H||%s||%an||%ai" --name-only
```

如果没有提交记录，告诉用户"最近 7 天没有提交记录"并停止。

再运行以下命令获取每次提交的改动量统计：

```bash
git log --since="7 days ago" --until="now" --all --no-merges \
  --format="%H" --shortstat
```

### 第 2 步：分析提交，划分前后端

对每个提交中涉及的文件，按以下规则判断前端/后端：

**前端标志（满足任一即判定为前端）：**

- 文件路径包含：`src/views/`、`src/components/`、`src/pages/`、`src/router/`、`src/store/`、`src/assets/`、`src/styles/`、`src/utils/`（前端项目）、`public/`、`static/`
- 文件后缀：`.vue`、`.jsx`、`.tsx`、`.svelte`、`.html`、`.css`、`.scss`、`.less`、`.styl`
- commit message 包含关键词：页面、组件、表单、弹窗、UI、交互、样式、布局

**后端标志（满足任一即判定为后端）：**

- 文件路径包含：`controller`、`service`、`repository`、`mapper`、`entity`、`model`、`dao`、`dto`、`vo`、`config`、`middleware`、`routes/`、`api/`
- 文件后缀：`.java`、`.py`、`.go`、`.rb`、`.php`、`.sql`
- commit message 包含关键词：接口、API、数据库、SQL、缓存、定时任务、服务、后端

**判断规则：**

- 同一提交同时包含前端和后端文件 → 拆分为两行记录
- 无法明确判断的 → 根据文件数量占多数的一方判定

### 第 3 步：归纳模块

根据以下信息自动归纳模块名称：

1. commit message 中的 scope（括号内的部分，如 `feat(用户管理):`）
2. 涉及文件的目录结构（取公共父目录作为模块名）
3. commit message 的 subject 语义（归类到相同业务领域）

模块命名规则：

- 使用简短的业务名称，如"用户管理"、"权限系统"、"报表模块"、"技术设计"
- 避免使用技术术语作为模块名（如"axios封装"应归入对应业务模块）
- 同一业务逻辑的提交归为同一模块

### 第 4 步：估算耗时

基于提交数量和改动量估算每个模块的耗时：

**基础规则：**

- 1 个人天 = 7.5 小时
- 1 小时 ≈ 0.14 人天

**估算方法：**

- 每次提交基础耗时 0.14 人天（约 1 小时）
- 根据 `shortstat` 的改动行数调整：
  - 改动 1~20 行：0.14 人天
  - 改动 21~50 行：0.28 人天
  - 改动 51~100 行：0.42 人天
  - 改动 100~200 行：0.56 人天
  - 改动 200 行以上：0.7 人天
- 设计类提交（仅文档、无代码）：0.14~0.28 人天

**总量调整：**

- 总计应 ≥ 5 人天
- 如果自动估算不足 5 人天，按比例等比放大到 5~6 人天范围
- 放大时保持各模块之间的比例关系不变
- 保留两位小数

### 第 5 步：撰写内容说明

用白话（通俗语言）概述每个模块的工作内容：

- 使用"完成了…"、"优化了…"、"修复了…"等直白表述
- 具体说明做了什么，不要用"相关工作"、"相关优化"等模糊措辞
- 如果只完成了部分功能，明确说明完成了哪一部分

**实事求是规则（必须遵守）：**

- git 上有提交记录的才能写进周报
- 只有设计文档提交 → 写"完成技术设计"，不能写"开发完成"
- 只有部分功能提交 → 写"完成 xxx 功能的 yyy 部分"
- 不虚构、不夸大任何没有提交记录的工作

### 第 6 步：输出周报表格

严格按以下格式输出：

```
| 模块 | 前端/后端：详细内容说明 | 花费时间 |
| :--- | :--- | :--- |
| 模块A | 后端：完成了 xxx 接口开发，支持 yyy 功能。 | 0.42 |
| 模块A | 前端：完成了 xxx 页面的表单开发和数据对接。 | 0.56 |
| 模块B | 后端：修复了 zzz 查询条件不生效的问题。 | 0.28 |
```

**格式要求：**

- 第二列必须以"前端："或"后端："开头
- 耗时列只写数字（人天数），不写单位
- 同一模块的前端和后端分行展示
- 按模块分组排列，不按时间顺序
- 表格最后附一行合计：`| **合计** | | **X.XX** |`

### 第 7 步：输出摘要

在表格之后，用 2~3 句话总结本周的整体工作重点。

## 注意事项

- 如果用户指定了时间范围（如"上周一到周五"），使用用户指定的时间替代默认的 7 天
- 如果用户指定了开发者（如"只统计我的提交"），在 git log 中加 `--author` 过滤
- 只统计当前项目的提交，不统计子模块或依赖项的提交
