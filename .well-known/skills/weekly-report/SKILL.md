# 周报生成

根据当前 Git 仓库指定时间范围内的提交记录，生成可复制到周报中的中文表格初稿。

## 使用场景

- 当用户要求写周报、生成周报、总结本周工作或汇报本周进度时使用。
- 默认统计当前分支最近 7 天的非 merge 提交。
- 如果用户指定时间范围、作者、分支或统计口径，优先使用用户指定条件。
- 不要把周报任务扩展成代码审查、发布说明或日报，除非用户明确要求。

## 工作流程

### 第 1 步：确认统计范围

1. 默认时间范围是最近 7 天：`--since="7 days ago" --until="now"`。
2. 如果用户说“本周”“上周”“上周一到周五”等，换算成对应日期范围。
3. 默认只统计当前分支；只有用户明确要求跨分支统计时才加 `--all`。
4. 如果用户指定开发者，在 `git log` 中加 `--author="<作者>"`。
5. 默认排除 merge commit；只有用户明确要求统计合并提交时才去掉 `--no-merges`。

### 第 2 步：收集提交数据

运行以下命令获取提交、作者、时间和涉及文件：

```bash
git log --since="7 days ago" --until="now" --no-merges \
  --date=iso-strict \
  --format="commit:%H%nauthor:%an%ndate:%ad%nsubject:%s" \
  --name-only
```

运行以下命令获取每次提交的改动量统计：

```bash
git log --since="7 days ago" --until="now" --no-merges \
  --format="commit:%H" --shortstat
```

命令调整规则：

- 用户指定时间范围时，替换 `--since` 和 `--until`。
- 用户指定作者时，追加 `--author="<作者>"`。
- 用户明确要求跨分支时，追加 `--all`。
- 如果当前目录不是 Git 仓库，说明无法读取提交记录，并请用户提供仓库路径或提交日志。
- 如果没有提交记录，输出“指定时间范围内没有提交记录”，然后停止。

### 第 3 步：归并提交

1. 去除重复提交，避免同一 commit 被重复统计。
2. 将同一业务模块、同一前后端类型的相关提交合并为一行周报记录。
3. commit message 太泛时，结合文件路径、目录名和改动统计推断实际工作内容。
4. 如果只能看出“文档、配置、依赖、脚本、测试”等工程性工作，按实际内容描述，不要包装成业务功能。

### 第 4 步：划分模块和前后端

模块名称优先按以下顺序确定：

1. commit message 中的 scope，例如 `feat(用户管理): ...`。
2. 涉及文件的业务目录或公共父目录。
3. commit subject 的业务语义。

模块命名规则：

- 使用简短业务名称，例如“用户管理”“权限系统”“报表模块”“技术设计”。
- 不要直接使用低层技术名作为模块名；例如“axios 封装”应优先归到对应业务模块或“前端工程”。
- 同一业务目标的多次提交归到同一模块。

前后端判断规则：

- 明确界面、页面、组件、样式、交互、表单、前端路由、前端状态管理，归为“前端”。
- 明确接口、服务、数据库、缓存、任务调度、权限校验、后端配置、数据模型，归为“后端”。
- 同一提交同时明确包含前端和后端工作时，拆成两行。
- `src/utils`、`config`、`api`、`model` 等名称可能在不同项目中含义不同，必须结合项目结构判断，不要只按关键词机械归类。
- 文档、脚本、测试、依赖、构建配置等非界面工作，如果无法明确归到前端，默认放在“后端”或对应工程模块下描述，但不要虚构后端接口。

### 第 5 步：估算耗时

直接估算每个模块的花费时间，作为周报初稿数值输出；用户会在复制到周报后自行微调。

估算规则：

- 1 个人天 = 7.5 小时。
- 每行记录至少 0.14 人天。
- 小改动、文案、配置、简单修复：0.14~0.28 人天。
- 普通功能、接口、页面、表单、数据对接：0.28~0.70 人天。
- 涉及多个文件、完整流程、联调或复杂逻辑：0.70~1.50 人天。
- 设计文档、方案整理、排查分析类工作：0.14~0.56 人天。
- 合并同一模块的相关提交后再估算，不要机械按提交数累加。
- 改动统计只作为参考；忽略生成文件、锁文件、格式化大面积变更对耗时的夸大影响。
- 保留两位小数。
- 不要为了凑满固定工时而放大耗时；只有用户明确要求按满周工时口径估算时，才可按用户指定总量调整。

### 第 6 步：撰写内容说明

用白话概述每行记录的工作内容：

- 使用“完成了...”“优化了...”“修复了...”“补充了...”等直白表述。
- 具体说明做了什么，不要写“相关工作”“相关优化”“功能完善”等模糊内容。
- 如果只完成了部分功能，明确写“完成了 xxx 的 yyy 部分”。
- 只有设计文档提交时，写“完成技术设计”或“补充方案文档”，不要写“开发完成”。
- 只有测试、配置、依赖或脚本提交时，按实际工程工作描述，不要包装成业务交付。

### 第 7 步：输出周报表格

严格按以下格式输出：

```markdown
| 模块 | 前端/后端：详细内容说明 | 花费时间 |
| :--- | :--- | :--- |
| 模块A | 后端：完成了 xxx 接口开发，支持 yyy 功能。 | 0.42 |
| 模块A | 前端：完成了 xxx 页面的表单开发和数据对接。 | 0.56 |
| 模块B | 后端：修复了 zzz 查询条件不生效的问题。 | 0.28 |
| **合计** | | **1.26** |
```

格式要求：

- 第二列必须以“前端：”或“后端：”开头。
- 耗时列只写数字，不写“人天”单位。
- 同一模块的前端和后端分行展示。
- 按模块分组排列，不按提交时间排序。
- 表格最后必须附合计行：`| **合计** | | **X.XX** |`。

### 第 8 步：输出摘要

在表格之后，用 2~3 句话总结本周整体工作重点。摘要必须基于表格内容，不要加入表格中没有体现的新事项。

## 规则

- 以 Git 提交记录为主要事实来源，不虚构没有提交支撑的工作。
- 如果用户额外提供了人工工作内容，可以合并进周报，但不要伪称它来自 Git 提交。
- 不要默认统计所有分支，避免把他人分支或历史分支的提交写进周报。
- 不要把 merge commit 当作独立工作量重复统计。
- 不要输出命令日志，最终只输出周报表格和摘要，除非用户要求展示分析过程。
- 如果提交信息和文件路径都无法判断业务含义，使用保守描述，不要臆测具体功能。

## 输出格式

```markdown
| 模块 | 前端/后端：详细内容说明 | 花费时间 |
| :--- | :--- | :--- |
| ... | ... | ... |
| **合计** | | **X.XX** |

本周主要围绕 xxx、yyy 和 zzz 展开，完成了 ...。
同时处理了 ...，提升了 ...。
```

## 验证

- 检查时间范围、作者过滤和分支范围是否符合用户要求。
- 检查提交是否去重，merge commit 是否按规则排除。
- 检查每条记录都能从 commit message 或文件路径找到依据。
- 检查第二列是否全部以“前端：”或“后端：”开头。
- 检查耗时合计是否等于各行数字之和，并保留两位小数。
- 检查摘要没有新增表格之外的工作内容。

## 注意事项

- 周报是给用户复制后再微调的初稿，耗时可以估算，但工作内容必须真实。
- 如果用户要求“按 5 天工作量写”，只调整耗时分配，不新增不存在的模块或成果。
- 如果仓库包含前后端多项目，必须先根据目录结构判断项目边界，再分类。
- 子模块、依赖目录、vendor 目录和构建产物默认不作为独立工作内容统计。
