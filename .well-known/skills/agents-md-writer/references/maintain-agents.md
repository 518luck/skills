# 维护 AGENTS.md 规则

本文件处理已有 `AGENTS.md` 的维护、补充、修正，以及把用户的概念、文档片段或 AI 常犯错转成可执行规则。

## 目标

维护已有 `AGENTS.md` 时，目标是最小修改、准确补充、避免重复。不要因为补一条规则重写全文。

## 维护前检查

1. 读取目标 `AGENTS.md`。
2. 如果目标在子目录，按 `references/nested-agents.md` 的父链读取规则读取祖先 `AGENTS.md`。
3. 读取与用户要求相关的项目文件或文档。
4. 判断用户输入类型：新增规则、修正旧规则、删除过时规则、概念转规范、AI 常犯错修复。
5. 找到最相关章节；没有合适章节时再新增小节。

## 概念转 AI 规范

用户给出的内容经常是概念、背景文档或对 AI 行为的抱怨。写入 `AGENTS.md` 前，必须先转成面向 AI 的行为规则。

转换维度：

- 触发条件：AI 在什么场景下需要遵守。
- 应做行为：AI 应该具体执行什么动作。
- 禁止行为：AI 不应该做什么。
- 判断标准：如何验证 AI 是否遵守。
- 适用范围：这是根目录全局规则，还是某个子目录局部规则。

改写模板：

```markdown
用户原话：<用户提供的概念、文档片段或问题现象>
规范化规则：<Always / Never / Prefer / Avoid / When... 形式的可执行规则>
适用范围：<全局 | 子目录路径>
放置章节：<AI 行为指令 | 代码风格 | 测试规则 | 架构约束 | 注意事项 | 其他章节>
验证方式：<如何判断 AI 已遵守>
```

写入原则：

- 优先把规则写成短句列表，每条只约束一个行为。
- 用 `Always` 表示强制动作，用 `Never` 表示禁止行为，用 `Prefer` 表示默认偏好，用 `Avoid` 表示不推荐做法，用 `When...` 表示场景化规则。
- 如果规则是为了防止 AI 重复犯错，优先放到 `AI 行为指令` 或 `注意事项`。
- 如果规则涉及命名、依赖、目录边界、测试方式、架构约束或安全要求，放到对应主题章节。
- 如果用户原话包含背景解释，只保留会改变 AI 行为的部分。

## 修改策略

- 保持原有结构和语气，除非结构本身导致规则无法遵守。
- 新规则放进最相关的现有章节。
- 一次维护只处理用户要求的规则，不顺手重构整份文件。
- 如果已有规则语义相同，优先改写或补强原规则，不新增重复条目。
- 如果新规则与已有规则冲突，先用项目文件或用户明确说明判断；仍不确定时提问。
- 如果父级规则应当更新，不要只在子目录补丁式绕过；提醒用户父级也需要维护。
- 删除规则前确认它确实过时、错误或被新规则替代。

## 转换示例

```markdown
用户说：AI 老是改无关文件。

写成规范：
- 聚焦当前任务，只修改完成用户请求所必需的文件。
- 不要进行无关重构、格式化或目录整理。
- 如果发现邻近代码有问题但不影响当前任务，只在最终说明中指出，不要直接修改。
```

```markdown
用户说：这里的接口概念比较复杂，要先看文档。

写成规范：
- 修改接口行为前，必须先阅读本目录的 `README.md` 和相关 API 文档。
- 不要仅凭函数名或类型定义推断业务语义。
- 如果文档与代码行为冲突，先说明冲突并询问用户，不要自行选择。
```

```markdown
用户说：这套组件有自己的设计语言，不要做成普通后台页面。

写成规范：
- 修改本目录 UI 前，先查看相邻页面和组件，延续已有视觉语言、间距、色彩和交互模式。
- 不要引入与现有设计系统冲突的通用后台布局。
- 新增组件时，优先复用本目录已有组件和样式 token。
```

## 验证清单

- 修改范围只覆盖用户要求的规则。
- 用户概念、文档片段或问题现象已转成触发条件、应做行为、禁止行为和验证方式。
- 新规则没有和现有规则重复。
- 子目录维护场景已经检查祖先 `AGENTS.md`。
- 没有删除仍然有效的项目特定规则。
- 没有引入无法从项目文件或用户明确说明推导出的命令或约定。
- 没有 `<占位符>`、`[待补充]`、`TODO`。
